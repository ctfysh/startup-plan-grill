# Stage 4: User Stories - 引擎与组织

Stage 4 是创业计划书中最"重"的阶段之一，覆盖增长、销售、团队、组织治理四条线，横跨三本书：Z2O（从0到1）、LS（精益创业）、LS2（精益创业2.0）。这三本书对组织的假设不同。Z2O 讲"小帮派、清晰角色、现金 vs 股票"的团队哲学，LS 讲"增长引擎、小批量、五问根因"的运营节奏，LS2 讲"创业部、创新核算、增长委员会"的制度化治理。这些思路可以互补，但在"创始人控制"和"制度驱动"之间存在张力。以下故事在需要处标注了这些张力点。

---

## 4A / 销售路径（Z2O）

### Q4.1 你的产品单价是多少？

**Story**: 用户是做校园二手教材 C2C 交易平台的项目，他说："我们的盈利模式是每笔交易抽佣 5-15 元，教材均价 30-50 元，抽佣比例大概 15-20%。"

**Demo**: Agent 说："这个价位学生自己就能决策，不需要走采购审批流程。但问题是，学生为什么要用你的平台而不是在闲置群直接交易？你多收的这笔佣金对应什么价值？" 用户想了想说平台提供交易保障、纠纷处理和物流对接。

**Acceptance**: 用户能清晰说出抽佣模式和金额，并且知道这个价位属于"个人可决策"。能说清楚佣金的对应价值。保障、物流、信任。

**Error**: 用户说"还没想好，大概抽 50% 吧"。Agent 回应："抽佣比例不能拍脑袋。定价本质是在'平台价值'和'用户接受度'之间找平衡。建议你先看看其他校园交易平台的抽佣比例。一般实物交易在 10-20% 之间。太高了用户会回到闲置群交易。"

**Edge**: 用户说"我们打算卖教材本身而不是做平台，一本赚 10 块"。Agent 应指出："自营卖教材和做 C2C 平台是两种完全不同的模式。自营需要压货、管库存、承担呆滞风险。平台只抽佣金，轻资产但需要解决信任问题。你选哪种决定了你的资金需求和扩张速度。"

---

### Q4.2 你打算怎么让第一批客户知道你的产品？

**Story**: 用户说："我打算先找各个学校的校园大使，让大使在班级群和朋友圈推广。先在武大做试点，跑通一个学校再复制到其他学校。"

**Demo**: Agent 说："校园大使地推是合理的获客路径。但一个学校需要几个大使？怎么考核业绩？按注册人数还是按交易笔数？第一批 5 个大使你亲自带，但到 50 个学校的时候谁去管？你有什么复制大使管理体系的计划吗？"

**Acceptance**: 用户能说清楚获客路径（校园大使/班级群/线上投放/线下摆摊），并且路径与单价匹配，低价走量。

**Error**: 用户说"先做个网站等学生自己来"。Agent 指出："大学生不会主动搜一个二手教材平台。他们目前已经习惯在闲置群交易了。你需要主动触达。去食堂门口摆摊、进班级群发红包、找学生会合作。"

**Edge**: 用户说"我不做地推，靠用户口碑传播"。Agent 肯定口碑的价值，但追问："口碑的前提是第一批用户用过觉得好。你不用地推，第一个用户怎么来？有没有一个最小启动方案？"

---

### Q4.3 你算过客户获取成本（CAC）和客户终身价值（CLV）吗？

**Story**: 用户说："算了一下，获取一个注册用户大概 8-15 元（地推传单 + 线上投放），每个学生在大学期间交易 2-4 次，佣金收入 20-50 元。CLV/CAC 大约 3。"

**Demo**: Agent 说："CLV/CAC 大概 3，在健康线以上，不错。但你的 CLV 假设每个学生会用你四年。你怎么保证学生不是用完一次就卸载？有留存数据支撑这个假设吗？"

**Acceptance**: 用户知道 CAC 和 CLV 的具体数字，CLV/CAC > 3，能说清楚 CAC 包含哪些成本项（传单/地推人力/线上投放/校园大使佣金），CLV 的计算基于什么留存假设。

**Error**: 用户说"CAC 大概 5 块吧，CLV 没算过"。Agent 说："不知道 CLV 就相当于你不知道每个学生到底值多少钱。建议至少先估算一下：一个学生在校期间交易几次？每次给你贡献多少佣金？先给个大概，再慢慢精修。"

**Edge**: 用户说"CAC 很低，但 CLV 也不高"。Agent 追问："确实。校园市场获客成本低但客单价也低。关键数字是 CLV/CAC 要大于 3，同时回收期要短。你的回收期是多久？学生第一单交易你就能收回获客成本的话，现金流就很健康。"

---

## 4B / 增长引擎（LS）

### Q4.4 你最适合哪种增长引擎？黏着式 / 病毒式 / 付费式？

**Story**: 用户说："校园二手教材天然适合同学推荐同学。一个学生用了觉得好，会拉室友一起来。所以应该是病毒式增长为主，辅以黏着式。"

**Demo**: Agent 说："病毒式就是每个用户带来几个新用户。你预计一个用户会推荐几个同学？推荐的转化率大概多少？如果目标是每学期翻倍增长，你算过需要病毒系数多少吗？"

**Acceptance**: 用户能选出与自己模式最匹配的增长引擎，并能说出为什么这个引擎适合、其他引擎为什么不合适。

**Error**: 用户说"三种都要"。Agent 回应："三种引擎的运营逻辑完全不同。黏着式要求低流失，病毒式要求高传播系数，付费式要求高 LTV/CAC。同时做三种等于一个都不专注。请你选一个最重要的作为起步。"

**Edge**: 用户说"病毒式获客 + 黏着式留存。我能不能双引擎？" Agent 指出："思路没错。校园场景天然适合病毒式。同学推荐同学。但黏着式的关键是把学生'每个学期回来交易'变成习惯。你打算怎么做留存？比如毕业季自动提醒卖书、下学期初提醒买书。"

---

### Q4.5 如果选病毒式，你的病毒系数估算过吗？大于 1 才能自增长

**Story**: 用户说："我们的邀请机制是每个用户推荐 5 个同学，大概 20% 会注册。那病毒系数大约是 1.0。"

**Demo**: Agent 说："1.0 刚好及格，大于 1 才能自增长。但你的 20% 转化率是假设还是实际数据？如果实际只有 10%，病毒系数就降到 0.5，增长引擎就停了。你打算怎么优化这个转化率？比如推荐双方各得 5 元红包？"

**Acceptance**: 用户知道病毒系数的计算公式（邀请数 x 转化率），能给出具体数值，并能判断是否大于 1。

**Error**: 用户说"没算过，但感觉挺多的"。Agent 说："感觉没有意义。公式很简单：每个用户邀请多少人乘以多少人注册。先估算一个数字，不行就做个小实验测一下。小于 1 等于零增长。"

**Edge**: 用户说"病毒系数现在是 0.5，但等用户量上来网络效应会拉高"。Agent 质疑："网络效应和病毒系数是两个东西。病毒系数是每个用户带来几个新用户，网络效应是用户越多平台越好用（教材种类更全）。它们可以同时存在，但不要混淆。"

---

### Q4.6 增长引擎一旦开启，天花板在哪？市场饱和时你做什么？

**Story**: 用户说："全国大学生大概 3000 万人，每人大学期间至少买 3-5 本二手教材。如果渗透率做到 5%，就是 150 万用户、每年几百万笔交易，够做很多年。"

**Demo**: Agent 说："150 万用户够了。但你增长引擎开起来后，是先做一所学校还是同时铺 10 所？不同学校的推广方式和成本完全不同。武大是综合性大学，隔壁华科是工科学校，学生的教材需求和推广方式可能完全不一样。天花板不是问题，问题是达到天花板之前你怎么保证引擎不停。"

**Acceptance**: 用户能估算市场的总容量，能判断自己增长引擎的上限，并且有"触顶之后怎么办"的思考。

**Error**: 用户说"没有天花板，市场无限大"。Agent 说："没有市场是无限的。如果你觉得没有天花板，说明你还没想清楚市场边界。请给一个具体的可服务市场数字。比如武汉所有高校的在校生人数。"

**Edge**: 用户说"天花板很远，但增长引擎可能先烧完钱"。Agent 说："你说到了关键点。增长引擎的可持续性取决于单位经济。与其担心天花板，不如先确认引擎本身效率够高。我们先回到 4.3 确认 CAC 回收期。"

---

## 4C / 小批量与自适应（LS）

### Q4.7 你是大批量一次性交付，还是小批量持续交付？

**Story**: 用户说："我们每两周发布一个小版本，持续迭代。这周加一个'扫码卖书'功能，下周优化搜索体验。"

**Demo**: Agent 说："两周一个版本不错。但你们能支持一天发布多次吗？如果不能，瓶颈在哪？技术架构还是人手不够？"

**Acceptance**: 用户能明确说出自己的交付节奏，并且知道小批量的好处（快速试错、降低风险），能区分"想做"和"能做到"。

**Error**: 用户说"等开发完所有功能一起发布"。Agent 说："这是典型的大批量思维。等全部做完才交付，风险极高。精益创业的核心就是小批量：做最小功能，让用户用，根据反馈改。比如先做武大一个学校的交易功能，验证了再加学校、加品类。"

**Edge**: 用户是做校园电商的，他说："我们涉及到线下物流，没办法每次迭代都改配送流程。" Agent 回应："物流确实不能像纯软件一样每周迭代，但你可以在 app 端做快速迭代，物流流程单独优化。精益原则同样适用，只是不同模块的迭代周期不同。"

---

### Q4.8 如果出问题，你有"五个为什么"根因分析的机制吗？

**Story**: 用户说："我们每次出线上事故都会复盘。比如上次用户付款后卖家不发货，最终发现是缺少自动取消订单的机制。"

**Demo**: Agent 说："有五问机制很好。但你们复盘后会跟进改进吗？上次取消订单的问题，有没有在系统里加了超时自动取消的逻辑？"

**Acceptance**: 用户有根因分析的意识，能举例说明，并且知道分析的目的是系统改进不是追责。

**Error**: 用户说"出了问题就修，没时间复盘"。Agent 说："不找出根因，同一个问题会反复出现。五次为什么不需要很长时间，但能帮你区分'表象原因'和'系统原因'。后者才是需要解决的根本问题。"

**Edge**: 用户说"我们每次都做五问，但发现最后还是人的问题。招了不靠谱的人"。Agent 指出："如果五问每次都停在'人不靠谱'，说明挖得还不够深。为什么招了不靠谱的人？是面试流程问题？还是急招降低标准？继续挖。"

---

## 4D / 团队基石（Z2O）

### Q4.9 你的联合创始人是怎么认识的？认识多久了？

**Story**: 用户说："我和联合创始人是大学同学，一起做过学生会项目，毕业后各自在互联网公司干了两年，现在一起出来做校园平台。"

**Demo**: Agent 说："认识很久了，这很好。彼得·蒂尔说创始人关系就像婚姻，和认识很久的人结婚比和陌生人结婚风险小得多。你们之间有没有发生过严重分歧？怎么解决的？"

**Acceptance**: 用户和联合创始人有足够长的共事历史，不是"刚认识就合伙"。双方有冲突处理的经验。

**Error**: 用户说"我们在一个创业活动上认识的，聊了一个月就决定合伙了"。Agent 警告："这就像在拉斯维加斯老虎机前遇到的陌生人直接结婚。不是不可能成功，但风险极高。建议先做一个小项目合作试试，或者约定一个短的试错期和退出条款。"

**Edge**: 用户是 solo founder。Agent 说："一个人创业不是不行，但 Z2O 强调创始团队应该有 2-3 个人互补。一个人要同时做产品、地推、融资，精力不够。你考虑过找合伙人吗？尤其是懂校园运营的人。"

---

### Q4.10 你们团队是全职还是兼职？

**Story**: 用户说："我和合伙人都全职在做，第三个人还在大厂上班，只能周末帮忙。"

**Demo**: Agent 说："你和合伙人全职是好的，但周末帮忙的合伙人是个问题。彼得·蒂尔说'要么上车要么下车'。兼职意味着他还没完全 commitment。建议给他一个截止日期做决定。"

**Acceptance**: 核心成员全部全职。如果有关键成员兼职，用户知道这是风险并有应对计划。

**Error**: 用户说"大家都是兼职先试试水"。Agent 说："全员兼职等于全员不投入。创业需要极度的专注和速度，兼职的人不会在凌晨两点爬起来处理交易纠纷。建议至少创始团队要全职。"

**Edge**: 用户说"我们都在兼职，因为还在读书/还要还房贷"。Agent 表示理解但追问："那你的资金计划是什么样的？有没有一个时间点大家能转全职？如果一直兼职，竞争对手全职在跑，你速度可能跟不上。"

---

### Q4.11 创始人的工资定多少？

**Story**: 用户说："我们两个人现在不拿工资，靠之前的积蓄生活。等拿到种子轮再给自己开最低工资。"

**Demo**: Agent 说："不拿工资是一个好信号。说明你们在乎公司未来的价值而不是眼前利益。积蓄能撑多久？如果种子轮不顺利，你们能坚持多长时间？"

**Acceptance**: 创始人工资接近最低水平，说明资金优先用在公司发展上。用户清楚这个水平的可持续性。

**Error**: 用户说"我给自己开 5 万月薪，跟之前在大厂一样"。Agent 说："彼得·蒂尔说过'政客不是创始人'。高薪创始人像职业经理人不像创业者。投资人看到创始团队高薪会非常警惕。建议降到够生活即可。"

**Edge**: 用户说"我有家庭负担，不能拿太低"。Agent 说："理解。这不是道德问题，是取舍问题。如果你必须拿高薪维持生活，那就接受这个事实，但要知道这会让你的公司在投资人面前显得不够 lean。也可以算一下你拿高薪等于公司每月多花多少钱，这笔钱如果用来招校园大使会不会更有价值。"

---

### Q4.12 你们用什么激励团队？现金还是股票？

**Story**: 用户说："公司还没注册，所以还没谈期权。但打算注册后设立期权池，早期员工都给期权。现金工资很低。"

**Demo**: Agent 说："尽早把股权结构定下来。口头承诺没有法律效力。到时候分股份可能扯皮。建议了解期权池和 vesting 的基本概念：4 年成熟期、1 年 cliff，这是标准做法。"

**Acceptance**: 用户知道股票/期权是早期团队的主要激励方式，有书面协议和 vesting 计划。现金和股票的搭配合理。

**Error**: 用户说"大家都是拿现金，没谈过股票"。Agent 说："早期公司现金有限，如果全发现金，一来现金流压力大，二来团队没有股权绑定，做两年就走了。建议学习一下期权池和 vesting 的基本概念。"

**Edge**: 用户说"我们给了 50% 股份给第一个员工"。Agent 说："慷慨是好事，但一次性给太多股权可能造成后续融资困难（投资人会担心股权结构）。建议采用 4 年 vesting + 1 年 cliff 的标准做法。"

---

### Q4.13 你的公司文化是什么样的？你希望招什么样的人？

**Story**: 用户说："我们的文化是'学生优先，信任为本'。我们自己做学生出身，知道学生最在意什么。招人标准就一条。能不能站在学生角度想问题。"

**Demo**: Agent 说："这个文化很务实。你面试的时候怎么测试这个能力？有没有具体的面试题或者情景题？文化不能只挂在墙上，要体现在招聘流程里。"

**Acceptance**: 用户说的不是空话套话，而是具体的文化特征。文化描述能指导招聘和日常决策。

**Error**: 用户说"我们有免费午餐和下午茶，大家关系很好"。Agent 说："免费寿司不构成文化。文化是你公司做什么和不做什么的原则。比如'学生第一'、'信任卖家和买家'、'快速解决问题'。福利只是锦上添花。"

**Edge**: 用户说"我还没想清楚文化"。Agent 说："早期文化是创始人价值观的自然延伸。你怎么做决定、你怎么对待学生用户、你容忍什么不容忍什么。把这些写下来就是文化。不需要很正式，但不能没有。"

---

### Q4.14 每个员工是只做一件事还是什么都要干？

**Story**: 用户说："现在两三个人，每个人都身兼多职。我既做产品又去食堂门口摆摊地推，合伙人写代码还管客服。"

**Demo**: Agent 说："早期重叠是正常的。但到 10 个人的时候，你打算怎么分工？是不是每个人有一条明确的职责线？角色重叠导致的最常见问题是，两个人都以为对方在做这件事，结果出问题了没人管。"

**Acceptance**: 用户知道早期可以身兼多职，但清楚什么时候需要界定角色。有理想到明确分工的时间点。

**Error**: 用户说"大家都随便干，谁有空谁做"。Agent 说："没有分工的团队效率极低。重要的事没人做，不重要的事三个人抢着做。建议至少每人有一块明确的 First Responsibility。"

**Edge**: 用户说"我们现在 3 个人，各做各的，但我觉得太割裂了"。Agent 说："人数少的团队既要分工又要协作。一个常见做法是：每个人有 primary 职责，但每周有个全员的 sync，让大家知道彼此在做什么。角色清晰加信息透明，不矛盾。"

---

## 4E / LS2 五原则与组织治理

### Q4.15 原则1 小团队：你的组织以多大规模的团队为基本工作单元？

**Story**（治理审计向）: 用户说："我们准备以 5-7 人组成一个小队，每个小队有明确的目标和自主决策权，不需要事事汇报。"

**Demo**: Agent 说："5-7 人是 LS2 推荐的小团队规模。你提到的'自主决策权'具体到哪些事？比如招人、预算使用、技术选型，小队自己能决定到什么程度？"

**Acceptance**: 用户有明确的团队规模规划，团队有实质性的决策自主权（不只是"名义上自主"）。知道小团队和传统部门的区别。

**Error**: 用户说"我们就是一个大团队，十几个人都向我汇报"。Agent 说："超过 10 人的团队如果没有分解成小团队，沟通效率会指数级下降。LS2 建议 5-7 人是理想单元规模。你考虑过拆分成 2-3 个小队吗？"

**Edge**: 用户说"我们没有小团队，但人也不多"。Agent 说："现在人不多可以扁平管理，但建议你提前设计好团队结构。等扩张到 20 人的时候再分，比现在先分好要痛苦得多。LS2 的核心理念是：小团队是组织的毛细血管，越早建立越好。"

**张力说明**: Z2O 强调创始人的绝对控制（帮派文化），LS2 强调小团队自治。这是张力所在。如果用户倾向于"所有事都要我拍板"，Agent 需要指出这种模式不可扩展。

### Q4.16 原则2 实验优先：重大决策前，你们是先讨论还是先做实验？

**Story**（治理审计向）: 用户说："我们现在的习惯是，有争议就做个小实验。比如两个设计方案拿不准？AB 测试走起。"

**Demo**: Agent 说："这是实验优先文化的标志。但你们实验失败了会怎样？有没有人因为实验失败被追责？如果团队害怕失败，实验文化就建立不起来。"

**Acceptance**: 用户不只是嘴上说"我们要做实验"，而是有具体的实验流程和失败容错机制。实验文化渗透在日常决策中。

**Error**: 用户说"我们开会争论很久，谁说服不了谁就老板拍板"。Agent 说："靠争论和老板决策有两个问题。第一浪费时间，第二建立在观点而不是数据上。建议改成：有争议就设计一个最小实验，用数据说话。"

**Edge**: 用户说"实验太慢了，我们等不起"。Agent 说："这是一个常见的误解。实验不是慢，而是避免了大赌注的失败风险。一个 AB 测试可能只需要几天，但一次错误的战略决策可能浪费几个月。你觉得哪个慢？"

### Q4.17 原则3 创业部：你的创新项目是由现有部门管理，还是由专门的创业部管理？

**Story**（治理审计向）: 用户说："我们现在所有项目都在一个团队里，既有老业务也有创新项目。感觉老业务总是吃掉创新项目的时间。"

**Demo**: Agent 说："这是典型的'老业务压倒新业务'。LS2 建议设立专门的创业部（Startup Department）来管理创新项目，和核心业务分离。你有考虑过把创新项目独立出来吗？"

**Acceptance**: 用户理解为什么要区分"现有业务管理"和"创新项目管理"，并且知道混在一起的风险（老业务永远优先）。

**Error**: 用户说"创业部是什么？没听说过"。Agent 说："这是 LS2 的核心概念。创新项目如果放在老业务部门里管理，会被 KPI 和日常运营压制。一个专门的创业部有独立的预算、独立的决策权，对创新项目全权负责。"

**Edge**: 用户说"我们公司太小，没必要分创业部"。Agent 说："你的逻辑是对的。3 个人的公司不需要部门分割。但建议你理解这个概念：不同性质的工作需要不同的管理方式。当你的公司发展到 30 人时，老业务的稳定性和新业务的探索性就应该分开管理了。"

### Q4.18 原则4 持续转型：你的组织有定期反思和调整方向的机制吗？

**Story**（治理审计向）: 用户说："我们每季度做一次战略复盘，看看哪些假设被验证了、哪些被证伪了、需不需要调整方向。"

**Demo**: Agent 说："季度复盘频率合理。但复盘有没有固定议程？结论怎么落地？有没有因为复盘而真正改变过方向？请举例。"

**Acceptance**: 用户不只是"做复盘"，而是把复盘当作组织学习的制度化流程。有案例证明复盘真正导致过方向调整。

**Error**: 用户说"我们的反思就是年底开个会总结一下"。Agent 说："一年一次不是持续转型，是年终总结。LS2 的持续转型要求更高频的反思。至少季度。建议你把战略复盘纳入固定日历。"

**Edge**: 用户说"我们每天都在忙，没时间做季度复盘"。Agent 说："最忙的时候恰恰是最需要复盘的时候。忙可能是在错误的方向上加速。LS2 有一句话：'走得快不如走得对。' 季度复盘不是可选项，是组织的刹车和方向盘。"

### Q4.19 原则5 创新核算：你的创新核算分三个层次吗？

**Story**（治理审计向）: 用户说："我们第一层次有数据仪表板，每天看活跃度和转化率。第二层次的商业案例分析也做了。第三层次 NPV 还在学。"

**Demo**: Agent 说："三个层次各走了一段，不错。第三层次 NPV 可以等你财务更成熟再做。现在最关键的还是让你们的第一层次仪表板跟踪先导指标（leading indicators），而不是事后指标。"

**Acceptance**: 用户理解三层创新核算的区别，至少前两层已在用，第三层知道什么时候需要。

**Error**: 用户说"创新核算是什么？和财务核算一样吗？" Agent 说："不一样。传统财务核算看过去（收入/利润），LS2 的创新核算是看未来。你的创新项目的进展不能只看收入，要用三个层次来评估：仪表板（先导指标）、商业案例（增长引擎）、净现值（含不确定性）。"

**Edge**: 用户说"我们只看收入。收入增长了就是在创新。" Agent 说："收入是滞后指标，等你看到收入下降已经来不及调整了。第一层次仪表板应该跟踪先导指标。比如注册率、激活率、留存率。这些才是创新的'心跳'。"

**张力说明**: LS2 的"三层创新核算"和对创新项目的精细化治理，与 Z2O 的"创始人直觉决定方向"可能存在冲突。习惯了靠直觉决策的创始人可能会觉得创新核算太 bureaucratic。Agent 需要指出：创新核算不是来替代直觉的，而是给直觉提供数据支撑。

### Q4.20 你的组织如何实现持续创新？靠制度而不是靠创始人灵光一现

**Story**: 用户说："现在基本靠我自己推动创新。我想到什么就让大家做什么。但我也知道这不是长久之计。"

**Demo**: Agent 说："你意识到问题所在了。靠创始人灵光一现的创新不可持续，因为你一个人不可能覆盖所有方向。LS2 建议把创新制度化。比如固定的创新时间（Google 的 20% 时间）、定期的 hackathon、或专门的创新预算。你有想过哪种形式适合你们吗？"

**Acceptance**: 用户认识到"创始人驱动"不可扩展，有至少一个制度化的创新机制构想。

**Error**: 用户说"我就是最大的创新发动机，有我在就够了"。Agent 说："假设你突然生病一个月，公司的创新就停摆了？好的创新机制应该让你的公司在没有你的时候也能持续产生新想法。"

**Edge**: 用户说"制度化会不会扼杀创新？" Agent 回应："这是个好问题。过度流程化确实会，但完全没有制度的创新等于随机事件。好的制度像护栏。给自由探索划定边界和资源，而不是给每一步都规定动作。"

### Q4.21 你设了创业团队作为基本工作单元吗？

**Story**: 用户说："我们准备成立两个创业团队（Startup Team），一个做新功能探索，一个做现有产品优化。每个团队有独立的目标和预算。"

**Demo**: Agent 说："两个团队分工清楚很好。但他们的决策自主权有多大？比如新功能探索团队能不能决定上线一个功能，还是要等老板批准？"

**Acceptance**: 用户不只是设立"名义上的创业团队"，而是给团队匹配了真实的决策权和资源。

**Error**: 用户说"我们有创业团队，但他们做什么还是要老板点头"。Agent 说："那就不叫创业团队，只是'小组'。LS2 的创业团队要有实质性的自主权。包括产品决策、实验设计、预算使用。没有自主权的团队只是在执行老板的想法。"

**Edge**: 用户说"我担心给了自主权他们做错决定"。Agent 说："一定会犯错。但小团队犯小错，比中央决策犯大错代价小得多。LS2 的逻辑是：用小的失败换取大的学习，而不是用控制避免小失败但冒大失败的风险。"

### Q4.22 你有没有一个专门的创业部来管理创新项目？

**Story**: 用户说："我们现在没设立创业部，但意识到创新项目总是被紧急业务打断。打算下个季度正式设立一个。"

**Demo**: Agent 说："从意识到行动是关键一步。创业部需要独立汇报线、独立预算、独立考核指标。你觉得你们能给到这些吗？还是只是一个'名义上独立'的部门？"

**Acceptance**: 用户理解创业部不是"在组织结构图上画一个框"，而是需要真金白银的独立资源和真正的汇报线分离。

**Error**: 用户说"创业部就是让一个 VP 兼管创新项目"。Agent 说："VP 兼管意味着创新只是他的副业。当核心业务出问题的时候，他一定会牺牲创新时间。创业部需要的是一个只对创新负责的人。"

**Edge**: 用户说"我们公司才 15 个人，现在谈部门是不是太早了"。Agent 说："15 个人确实可以更灵活。但功能分离可以提前做。比如指定 2-3 个人只做探索型项目，不背负 KPI。关键是'让你的创新工作不受老业务干扰'，而不是纠结于部门名称。"

### Q4.23 你如何做到"大处着眼，小处着手，迅速扩大"？

**Story**: 用户说："我们的愿景是改变大学生买二手教材的方式，但起步只做武大一个学校的教材交易。先跑通一个学校的模型再复制到全国。"

**Demo**: Agent 说："这就是典型的'小处着手'。一个学校、一个场景、深度验证。你计划怎么从第 1 个学校跑到第 100 个？复制的关键假设是什么？是校园大使模式可复制，还是技术平台可标准化？"

**Acceptance**: 用户有清晰的"从小到大的路径规划"。从哪个点切入、验证后如何复制、复制的条件和节奏是什么。

**Error**: 用户说"我们想直接做大。融资后全国铺开"。Agent 说："直接铺开的风险极大。你连一个学校的模型都没验证过，凭什么证明这个模式在其他学校也能跑通？LS2 建议先从最小单元开始，跑通经济模型后再放大。"

**Edge**: 用户说"我们从小处着手了，但不知道什么时候该扩大"。Agent 说："阶段转换的触发条件要提前设好。比如'单校月交易额达到 10 万且获客成本降到 5 元以下时，开始扩大'。没有明确的 trigger，你可能会在小处停留太久，也可能在没准备好时就扩张。"

### Q4.24 你的创新核算分三个层次吗？仪表板 / 商业案例 / 净现值

**Story**: 用户说："我们仪表板层级建好了。每天看新注册、激活率、7 日留存。第二层商业案例在用，第三层 NPV 还没做。"

**Demo**: Agent 说："第一层最好，第二层需要加强。商业案例不只是算收入。你算了增长引擎的可持续性吗？比如病毒系数趋势、CAC 回收期变化？这些是判断增长引擎是否健康的核心指标。"

**Acceptance**: 用户不止知道三层概念，而且已经在实践中使用前两层，第三层有了解。

**Error**: 用户说"我们有财务报表就够了，不需要三层次"。Agent 说："财务报表是滞后指标，告诉你上个月赚了还是亏了。创新核算的仪表板是先导指标，告诉你下个月会怎么样。两者都重要，但如果你只看财报，等于开车只看后视镜。"

**Edge**: 用户说"NPV 太复杂了，我不想碰"。Agent 说："第三层次 NPV 确实是最复杂的。如果你现在不需要用它来做投资决策，可以跳过。但建议至少知道它的用途。当你要在多个创新项目之间分配预算时，NPV 是最好的比较工具。"

### Q4.25 你设立了增长委员会吗？

**Story**: 用户说："我们每月的项目评审会基本就是我在看所有人的进展，然后我决定哪个继续哪个砍掉。"

**Demo**: Agent 说："现在人少你可以一个人决策。但 LS2 建议设立增长委员会。一个跨部门的评审机构，不是老板一个人说了算。增长委员会的职责是：评审创新项目、分配资源、决定 killing。这样有两个好处。决策更民主，也让你不会因为是自己的项目而不舍得砍。"

**Acceptance**: 用户理解增长委员会的价值，不把它当作"多余的官僚流程"，而是资源配置和风险控制的关键机制。

**Error**: 用户说"不需要委员会，我一个人拍板更快"。Agent 说："创始人拍板的效率确实高，但也有两个问题。第一你个人有认知盲区，第二你很难客观地 killing 自己的项目。增长委员会不是来拖慢你的，是来帮你做出更理性的资源分配决策。"

**Edge**: 用户说"我们设了委员会，但最后所有人都不敢得罪老板，还是老板说了算"。Agent 说："这是最常见的失败模式。名义上的委员会，实质上的独裁。打破这个局面需要：一、明确的投票规则；二、委员会成员有不同背景（不仅仅是高管）；三、老板也遵守委员会决定。"

## 4F / 三阶段转型路线图（LS2）

### Q4.26 第一阶段 临界规模：从几个团队开始试点？

**Story**: 用户说："我们计划先从两个创业团队试点，一个做用户增长实验，一个做新功能探索。先用三个月跑通流程。"

**Demo**: Agent 说："两个团队起步合理。太少不具备参考性，太多管不过来。三个月是一个好的试运行周期。试点的成功标准是什么？什么情况下你觉得'这个模式可行'，可以在更多团队推广？"

**Acceptance**: 用户知道先从少数团队开始试点，有明确的时间线和成功标准。

**Error**: 用户说"直接全公司推，不用试点"。Agent 说："LS2 强烈建议不要一步到位。因为创业团队的工作方式和传统部门完全不同，你需要先在小范围内验证这种模式在你公司是否可行。先试点，再推广。"

**Edge**: 用户说"我们所有团队都想参加试点，怎么选？" Agent 说："选最有意愿、最理解创新理念的团队。他们有热情，也更能承受失败。第一波试点的目标不是覆盖多少人，而是跑通方法和积累经验。"

---

### Q4.27 第二阶段 扩大规模：怎么从试点推广到全公司？

**Story**: 用户说："如果试点成功，我们计划每季度增加 2-3 个创业团队。同时把试点中总结出的最佳实践写成标准化流程。"

**Demo**: Agent 说："标准化流程是个好主意。但要注意。标准化不等于 rigid。扩大规模最难的是保持创业团队的自主性。你有想过怎么在扩大的同时防止流程僵化吗？"

**Acceptance**: 用户有扩大规模的具体计划。节奏、方法、如何保持质量。知道扩大不只是"招更多人"。

**Error**: 用户说"试点成功了就照搬不就行了？" Agent 说："扩大规模往往比试点更难。因为人多了、文化稀释了、流程僵化了。LS2 建议在扩大过程中不断复盘，一边扩大一边调整方法，而不是机械复制。"

**Edge**: 用户说"如果扩大后质量下降怎么办？" Agent 说："要提前设好质量护栏。比如每个新成立的创业团队前三个月要有导师（从试点团队抽调）。同时用创新核算的第一层次仪表板监控每个团队的先导指标，一旦偏离及时干预。"

---

### Q4.28 第三阶段 深层机制：法务/财务/IT/HR 如何支持创新？

**Story**: 用户说："目前法务、财务、IT 都是按照传统业务设计的。审批流程长、预算固定。如果要支持创新，这些部门需要改变。"

**Demo**: Agent 说："你看到了核心瓶颈。LS2 认为第三阶段最难的不是创新团队本身，而是支持部门如何调整。具体来说。法务能不能加快合同审批速度？财务能不能给创新团队单独的灵活预算？IT 能不能快速配置实验环境？你有计划推动这些改变吗？"

**Acceptance**: 用户认识到第三阶段的瓶颈不在创新团队，而在支持体系。有推动职能部门改革的具体想法。

**Error**: 用户说"职能部门支持创新？不干涉就不错了。" Agent 说："这就是问题所在。如果财务按传统 KPI 考核创新项目，法务按传统流程审批每个实验，IT 按标准排期部署环境，你的创业团队根本跑不动。第三阶段的核心就是让职能部门变成创新加速器而不是刹车。"

**Edge**: 用户说"职能部门抵制怎么办？他们觉得自己本职工作已经很忙了。" Agent 说："这是常态。LS2 建议给职能部门设立'创新支持 KPI'。比如财务对创新项目的审批速度、IT 对实验环境的部署速度。定义清楚职能部门在创新中的角色和责任。"

**张力说明**: LS2 的第三阶段本质上是在改造传统组织的运作方式，这与 Z2O 的"创始人就是规则"存在深层张力。如果创始人习惯了"我说了算"，把权力下放给职能部门和创新团队可能会感觉失控。Agent 需要在用户表现出这种犹豫时指出：制度化的组织比你一个人更强大。

---

### Q4.29 你设立"创新沙盒"了吗？

**Story**: 用户说："我们还没有创新沙盒，但听了之后觉得很有必要。比如我们的产品发布有严格的审批流程，如果创新团队每次实验都要走一遍，速度就太慢了。"

**Demo**: Agent 说："创新沙盒的精髓就是'在规矩之内打破规矩'。划定一个创新团队可以豁免常规流程的边界，比如额度限制、时间限制、用户数量限制。你们可以先从一个沙盒开始。比如允许创新团队在 100 个用户范围内做实验，不走完整发布流程。"

**Acceptance**: 用户理解创新沙盒的概念，能想到自己公司哪些流程需要沙盒豁免，并且愿意给创新团队一定的自由度。

**Error**: 用户说"什么沙盒？我们一视同仁。所有人都走一样的流程。" Agent 说："用管理成熟业务的流程来管理创新项目，是最常见的创新杀手。创新需要实验空间。沙盒就是一个受保护的空间，让创新团队在一定范围内不受常规流程约束。"

**Edge**: 用户说"给了沙盒他们乱来怎么办？" Agent 说："沙盒是有边界的。不是无限自由。常见边界包括：预算上限（比如 10 万以内）、时间窗口（比如 3 个月）、用户规模（比如 1000 人以下）、风险等级（不允许涉及合规红线）。设定明确边界后，在边界之内给他们最大的自由。"
