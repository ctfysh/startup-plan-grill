# Stage 3: User Stories - 精益验证

> **跨书张力提示**: Stage 3 由 Lean Startup（LS）主导。这里是与 Zero to One（Z2O）哲学冲突最尖锐的区域。Z2O 在 Stage 0 让用户相信"我找到了一个秘密，我要垄断它"，LS 却说"你的假设很可能是错的，去测试它"。Agent 在 Stage 3 中的角色不是打击信心，而是温和地挑战确定论思维，帮用户从"我知道答案"切换到"我有个假设，让我验证一下"。

---

### Q3.1 精益创业的第一个概念是"经证实的认知"（validated learning）。你怎么定义"学到了"？

**Story**: 用户说："我用一周时间做了个小程序，让武大学生能买卖二手教材。上线一周已经有 50 个人注册了。这就是学到了吧？"

**Demo**: Agent 先肯定了速度，然后追问："精益创业说的'经证实的认知'，不是'做了就知道了'，而是你在做之前能明确说出，我要学到什么？学到什么程度算'学到了'？比如，50 个注册用户里有多少完成了交易？你是想看到 10% 的发布转化率，还是想确认学生真的愿意为一本二手教材等两天物流？"

**Acceptance**: 用户能说出一个可验证的具体标准，比如"我要看到至少 10% 的注册转化率"或"我要跟 5 个目标用户聊完，他们都愿意付费"。标准越具体，越容易判断"学没学到"。

**Error**: "产品做好了自然就知道好不好。" 这等于没有学习目标。Agent 回："如果没有预设学习目标，产品上线后你面对一堆数据，你也不知道该看哪个指标。我们要先定义'学到了'的标准，不然做完了也不知道下一步去哪。"

**Edge**: 用户说"我学会了就是用户说喜欢。" "喜欢"太模糊了。Agent 追问："用户说喜欢，和用户愿意掏钱，是两回事。你说的'喜欢'是指什么？好评？复购？推荐给朋友？"

---

### Q3.2 你这个商业模式中，最冒险的假设（leap-of-faith assumption）是什么？

**Story**: 用户说："最大的风险……闲鱼吧。如果闲鱼也搞一个校园二手教材专区，我们就没有优势了。" 用户习惯性地把风险往外推，外部竞争。

**Demo**: Agent 没有否定，但把问题转了回来："闲鱼确实是潜在的竞争对手，但精益创业问的是'最冒险的假设'。不是外部风险，而是你商业模式里那个'如果不成立，整个生意就完蛋'的核心信念。比如，你的核心假设是'学生愿意在一个专门的二手教材平台上买卖，而不是直接在闲鱼上挂'。如果这个假设错了，你的平台做得再好也没用。你能接受这个假设可能错吗？"

**Acceptance**: 用户能识别出一个内在于商业模式的假设，而不是泛泛的外部风险。比如"我假设学生愿意为了更快找到教材而使用专门平台"或"我假设 5-6 折的价格足够吸引卖家来我的平台而不是去闲鱼"。这个假设如果错了，整个模式就站不住。

**Error**: 用户一直说外部风险（竞争、政策、经济）。Agent 温和但坚定地收回："我们先把外部风险放一边，聚焦在你自己画的商业模式上。画布 9 个格子里，哪个格子里的假设是你最不确定的？"

**Edge**: 用户说"我觉得没有假设，我确定学生需要这个。" 这是 Z2O 后遗症。Stage 0 的"秘密"信念太强了，用户已经说服了自己。Agent 回："明白，你已经做了很多思考。但咱们把'确信'暂时放下，问一个问题。如果非要找一个最可能出错的地方，假设这个生意最脆弱的地方在哪里？" 把"信念"转化成"脆弱性"，心理阻力会小很多。

---

### Q3.3 区分一下你的价值假设和增长假设分别是什么？

**Story**: 用户说："两个都重要。我想验证学生愿不愿意花 5-6 折买二手教材，也想知道他们会不会推荐给同班同学。不能只看一个吧？" 用户想同时覆盖一切，典型的创业者思维，不做取舍。

**Demo**: Agent 笑着认同："你说得对，两个都重要。但现在我们要分清楚，哪个是前提？从逻辑上说，如果学生不愿意为二手教材付钱，增长就是空的。你推广一个没人想买的平台，增长越快死得越快。先确认价值假设：你最有信心的证据是什么，证明学生真的愿意在这个平台上花钱买旧书，而不是去闲鱼比价？"

**Acceptance**: 用户能给出一个具体的价值假设验证方法。比如"我打算先让 50 个学生上架教材，看有多少本被买走。如果有 10% 以上成交，就算验证了价值假设。" 区分清楚价值假设在先，增长假设在后。

**Error**: 用户说"先增长再说，有了流量自然有成交。" Agent 提醒："这是一个很常见的陷阱。流量来了，但学生不愿意在专门平台上交易，你的增长只会放大亏损。先看有没有人愿意在平台上买卖，再考虑怎么让更多学校加入。"

**Edge**: 用户说"我的平台对学生免费，靠广告赚钱，价值假设不适用吧？" Agent 追问："好，那价值假设变成了'学生会在平台上持续活跃浏览教材信息'，这样广告才有价值。你能验证这个使用黏性吗？" 价值假设适用于所有模式，只是形式不同。

---

### Q3.4 你做过类比分析吗？有没有类似产品/模式可以参考？

**Story**: 用户说："校园二手教材这个模式挺新的，没看到谁专门做这个。不好类比吧？" 语气里带着一点"我是创新者"的骄傲。

**Demo**: Agent 没有反驳，而是引导："没有完全一样的产品，不代表没有可类比的模式。你的用户现在怎么处理旧教材的？闲鱼上挂、校内 QQ 群转让、或者干脆扔掉。这些都是替代方案。再看国外，Chegg 就是做教材租赁和买卖的成熟模式。你可以在两个维度做类比：闲鱼告诉你中国学生的二手交易习惯，Chegg 告诉你教材这个品类的商业模式。"

**Acceptance**: 用户至少能找到一个可类比的对象，即使不是直接竞品。比如"学生现在用 QQ 群买卖教材，效率很低但也能用"或"有点像 Chegg 但我们是 C2C"。能说出"我们的不同在于专注校园和教材品类"就算合格。

**Error**: "没有可比的产品。" 这种回答通常意味着用户没有认真想。Agent 继续追问："那学生现在没有这个平台的时候，怎么解决教材问题的？用闲鱼？用 QQ 群？咬咬牙买新书？那些'咬咬牙买新书'的人就是你的目标用户。你的平台要明显好到让他们不想再花大价钱买新书。"

**Edge**: 用户说"我们像闲鱼但又不是闲鱼。" 太模糊了。Agent 回："闲鱼做的很多。全品类二手、社交互动、同城面交。你说像闲鱼是指哪一层？用户群体、交易模式还是推广渠道？越具体，类比越有用。"

---

### Q3.5 什么是最小可行产品（MVP）来测试你的核心假设？

**Story**: 用户说："MVP 啊，我想做一个完整的小程序。用户能注册、上架教材、搜索、下单、支付。大概需要开发 2 个月吧。" 用户的 MVP 已经朝着"简化版闲鱼"去了。这是最常见的误解。

**Demo**: Agent 没有直接说"不对"，而是用一个问题来检验："好，假设 2 个月后你发布了这个完整小程序，它的核心目标是验证什么假设？注意，MVP 不是为了上线产品，而是为了用最少成本测试你那个最冒险的假设。学生愿意在专门的教材平台上交易。你能不能设计一个更小的实验？比如一个微信群加 Google 表单，学生发教材信息、你人工在群里撮合交易，花一周时间，看看能不能促成第一笔成交？"

**Acceptance**: 用户重新理解了 MVP 的本质。不是最小功能产品，而是最快学习实验。能主动说出"原来 MVP 可以不用写代码"就明白了。设计方向从"哪些功能不能少"转为"最小的实验能学到什么"。

**Error**: "MVP 就是先做一版，后面再迭代。" 这是最常见的误解，把 MVP 当成了 v0.1。Agent 纠正："迭代思维是对的，但 MVP 不是'简陋版完整产品'，而是'验证单个假设的实验'。如果你做的功能同时测试三个假设，一旦失败你根本不知道哪个假设错了。"

**Edge**: 用户说"交易平台必须做完整才能测试，因为学生不会用半成品交易。" Agent 回："那这个信念本身就可以测试。你建一个微信群，在群里人工匹配买卖双方，学生在群里完成交易。他们能看出来背后没有自动化系统吗？试试就知道了。"

---

### Q3.6 这个 MVP 是哪种类型：视频 MVP / 贵宾式 MVP / 人工替代型 MVP / 残缺型 MVP？

**Story**: 用户说："那就做一个小程序 MVP 吧，让学生能注册和上架教材。" 用户默认选了开发量不小的类型，因为他做过技术，本能反应是做产品。

**Demo**: Agent 先肯定了这个方向，然后列出了选项："小程序是一种，但你的核心假设是'学生愿意在这个平台上交易'，贵宾式 MVP 可能更快。你人工帮第一个买家找书、人工协调交易，0 行代码验证有没有真实需求。或者先建一个微信群，学生发买书卖书信息，你手动匹配。你的核心假设是什么？选了最匹配的类型吗？"

**Acceptance**: 用户能根据自己要验证的假设选择 MVP 类型。验证"学生愿意交易"选贵宾式（人工撮合），验证"学生会频繁使用"选残缺型（先做一个最简单的上架功能）。类型和假设之间有明显逻辑关系。

**Error**: "我不确定用哪种，你觉得哪种好？" 用户把选择权推给 agent。Agent 不直接给答案，而是反问："你想验证价值假设还是增长假设？价值假设适合贵宾式或人工替代型 MVP，增长假设适合残缺型 MVP。你自己选。哪个实验成本更低、风险更小？"

**Edge**: 用户说"我能不能同时做两种 MVP？" 精力分散是大忌。Agent 回："可以，但建议先做最便宜的那个。如果最便宜的实验就得出了明确结论，你就不需要做第二个了。先用微信群验证有没有真实需求，如果有，再考虑要不要做小程序。"

---

### Q3.7 MVP 再砍一刀还能去掉什么？（目标：砍到不能再砍）

**Story**: 用户说："MVP 已经很小了。注册、上架教材、搜索、下单。不能再砍了，再砍就不是一个交易平台了。" 用户心里的 MVP 下限还是有四个功能。

**Demo**: Agent 开始扮演"砍刀"角色："好，假设只能保留一个功能，哪个保留？如果'上架教材'是核心验证功能，那注册能不能先砍掉？把上架做成一个 Google 表单提交，不需要注册。搜索能不能直接用 Excel 表格分享给群成员？'不能再砍'常常只是不想动自己已经想好的方案。试试看，再砍一刀，还剩什么？"

**Acceptance**: 用户经过三轮"再砍一刀"后，最终剩下的核心功能跟最初说的"四个必须"完全不同。用户理解了"核心假设"和"锦上添花"的区别，最终 MVP 可能只是一个表单加人工匹配。

**Error**: "砍不了了，这四个功能是一个整体。" Agent 不纠缠，换一个角度："好，那这四个功能如果只能选一个来验证假设，你选哪个？选好了，另外三个就不是 MVP 的一部分，是以后的事。"

**Edge**: 用户砍到最后说"那 MVP 就只是一个微信群？" 语气有点失望，觉得太小了没面子。Agent 回："微信群也行。精益创业的案例里，Dropbox 的 MVP 只是一个视频，视频里的人甚至没写代码。如果你的假设能被一个微信群验证，那写小程序就是浪费。"

---

### Q3.8 你做 MVP 需要多久？（理想：天级，不是月级）

**Story**: 用户说："一个月吧。做一个简单的校园二手小程序，能跑通教材上架和下单流程，一个月已经很快了。" 语气里觉得自己已经很克制了。

**Demo**: Agent 没有否定，但追问了时间成本背后的风险："一个月里你投入了开发资源，如果 MVP 上线后发现学生根本不愿意在专门平台上交易，这一个月的代码全部白写。如果改成三天就能上线的实验，一个微信群加人工撮合，你同样能验证假设，但失败成本降了 90%。你的一周版本长什么样？"

**Acceptance**: 用户能把 MVP 时间压到"天数"而非"月数"。主动说出"其实三天内我就可以建个群、发个表单、手动帮学生匹配交易"，这说明用户理解了时间本身就是最大的风险变量。

**Error**: "最少也得一个月，我找外包做小程序最快就是这个速度。" Agent 回："外包开发一个 MVP，本质上是把你验证学习的时间外包出去了。你等了一个月才知道假设对不对。有没有不用等外包也能做的实验？比如先建群手动撮合，有需求了再找外包开发？"

**Edge**: 用户说"我一天就能做出来，建个微信群发个通告就够了。" 这是好回答。Agent 鼓励并进 3.9："这个速度很好。那在你上线这个最快 MVP 之前，我们先设好衡量标准。什么算验证通过？"

---

### Q3.9 STEP 1 基准线：MVP 上线前，你当前的核心指标是什么？

**Story**: 用户说："现在 app 还没上线，用户数为零。上线后只要有人上架教材就算成功吧？" 用户用"有人用"当成功标准，太模糊了。

**Demo**: Agent 把标准收紧："基准线是零没错，但我们需要的不是'有人上架就行'，而是一个具体的数字标准。比如上线第一周你期望 30 本书上架、10 笔交易完成。你的校园里有几千个学生，你打算怎么推广、预计多少本教材能上架？"

**Acceptance**: 用户能给出一个有依据的具体数字基准，而不是模糊的有或无。比如"我打算在武大食堂门口发传单，第一天预计 50 个人扫码进群，一周内预计 30 本教材上架、10 笔交易完成。" 用户明确知道现在的基准是零，也知道第一个目标数字是什么。

**Error**: "先上线看看数据再说。" Agent 不同意："如果不知道目标，数据不会告诉你下一步怎么做。假设上线后群里有 5 个人上架了教材，你觉得是成功还是失败？没有目标，你怎么判断？至少拍一个数字，哪怕不准，它也给了你一个判断基准。"

**Edge**: 用户说"我不是互联网产品……" Agent 回："校园二手教材平台虽然是小程序，但核心指标也是一样的。交易量、转化率。选一个你能追踪的计量指标作为基准线：本周教材上架数？本周成交数？进群人数？"

---

### Q3.10 STEP 2 调优：你预期实验后哪些指标会变好？好多少算"成功"？

**Story**: 用户说："当然是越多越好，上架 100 本成交 100 本就完美了。" 半开玩笑，但暴露了用户没有认真想"什么算好"。

**Demo**: Agent 笑着把问题拉回现实："完美当然好，但现实一点。假设你发了 500 份传单、100 个人扫码进群，你心里觉得至少几个人上架教材才算'这个方向对了'？注意，我不是问你理想的目标，我问的是，如果低于这个数字，你会开始担心。那个数字是多少？"

**Acceptance**: 用户能说出一个具体的数字目标和背后的推理逻辑，比如"我希望群里有 30% 的人发布教材，因为目标用户是快毕业的高年级学生，他们最急于处理旧书。" 不要求数字精准，但要求有思考过程。

**Error**: "我觉得能有 50 本上架。" 随意报了个数。Agent 追问："这个 50 本怎么来的？你是从传单转化率推算的，还是直觉？如果只是拍脑袋，实验结束后你很难判断。到底是假设错了，还是你的目标定得太高了？"

**Edge**: 用户反问"那你觉得多少合理？" Agent 不替用户回答，而是引导："我没有你的学校数据，所以不知道。但我可以帮你拆解思考。你的目标用户是哪个年级？你会去哪里推广？一次传单能覆盖多少人？"

---

### Q3.11 STEP 3 转型还是坚持：如果指标没达到预期，你何时决定转型？

**Story**: 用户说："如果第一周没什么人交易，那就继续推呗，多发传单、多做活动。创业哪有一次成功的。" 用户把"坚持"当成了美德，拒绝面对转型的可能性。

**Demo**: Agent 先认可了用户的韧性，然后巧妙地把问题从"要不要转"转向了"何时转"："坚持是好品质，但提前设一个'红灯数字'会更理性。比如，如果 MVP 上线两周后总教材上架量不到 20 本、成交不到 5 单，你会不会开始思考。也许学生真的不习惯专门平台，而是习惯在闲鱼或 QQ 群里解决？提前设好这个条件，比数据不好时再纠结要理性得多。"

**Acceptance**: 用户能给自己设一个"如果……就转型"的触发条件。比如"如果前 200 个进群的学生里教材上架率低于 10%，我会重新审视这个平台对学生到底有没有价值。" 这个条件必须是可量化的、有时间限制的。

**Error**: "我不会放弃的，坚持下去总会有转机。" Agent 不跟用户硬碰，而是换个角度："坚持一件事不等于坚持一种方法。如果学生就是不愿意在专门平台交易，坚持下去只是更快地撞墙。设一个触发条件不是为了'放弃'，而是让你知道什么时候切换方法更聪明。"

**Edge**: 用户说"如果数据不好，我就换另一个学校试点。" 太随意的转型计划也很危险。Agent 回："好，那你提前想过如果武大不行还有其他选择吗？不是临时拍脑袋想一个，而是提前列出可能的 2-3 个方向？"

---

### Q3.12 你跟踪的是可执行指标还是虚荣指标？

**Story**: 用户说："我关注上架教材的数量，上周有 30 本上架了。这个数据不错吧？" 用户选了一个最常见的虚荣指标而不自知。

**Demo**: Agent 没有打击，而是用一个例子来区分："上架数是典型的虚荣指标。它让你感觉不错，但它不告诉你这些书有没有真的卖出去。可执行指标是'交易完成率'。上架的书里有多少比例最终成交了。如果上架数很高但交易完成率只有 10%，你真正的问题不在供给而在撮合效率。你目前能追踪到的是哪个层次的指标？"

**Acceptance**: 用户能区分"看起来好"和"能指导行动"的指标区别，并选出一个可执行指标。比如"我明白了，我应该关注的是教材交易完成率，而不是上架教材的总数。"

**Error**: "两个都看不行吗？" Agent 回："你当然可以两个都看，但你必须选一个作为决定是否转型的依据。那个指标必须是可执行的。它失败了你就明确知道该做什么改变。上架数下降了，你不知道是推广问题还是学期末没人卖书了。但交易完成率下降了，你知道该调整匹配效率或信任机制。"

**Edge**: 用户说"除了交易完成率还有别的可执行指标吗？" Agent 回："当然有。比如'发布到成交的平均天数'。如果这个数字太长，说明匹配效率低。或者'重复购买率'。如果一个学生卖了一次又来卖第二次，说明他对平台满意。选一个跟你最冒险的假设直接相关的指标。"

### Q3.13 你的第一个 BML 循环（从 idea 到学习结论）最短能做到多短？

**Story**: 用户说："一个循环……大概一个月吧。建个群、等人上架、等人买、看看效果，差不多一个月。" 用户脑子里还是相对慢的节奏。

**Demo**: Agent 引导："艾瑞克·莱斯说，BML 循环的速度决定了你的学习速度。一个月一个循环，一年你只能学 12 次。但你能不能把第一个循环缩短到一周？比如这周一你去食堂发传单、周二就有人扫码进群、周三有人上架教材、周五成交第一单。从发传单到确认'有人愿意在平台上交易'，你只花了 5 天。这样一周你就完成了从构建（建群）到衡量（多少人上架、成交）到学习（平台模式是否成立）的完整循环。"

**Acceptance**: 用户能把 BML 循环时间压到一周以内，甚至说出"其实 3 天就够了。第一天发传单，第二天群里有人问价，第三天人工撮合成交第一单。" 用户理解了循环速度本身就是竞争优势。学得快的团队永远比学得慢的团队先找到正确答案。

**Error**: "慢点没关系，品质更重要。" Agent 回："品质当然重要，但如果你在验证阶段追求高品质小程序开发，你可能花了很多时间做一个没人愿意用的教材交易平台。验证阶段速度优先，证明了需求之后，再追求品质。"

**Edge**: 用户说"我的问题不是速度，而是学生只在学期末才卖书。" Agent 回："那这个 BML 循环就要拉长到学期周期？不一定。你可以换个角度验证。不验证'卖书'，而是验证'学生愿意在平台上看教材信息'。学期初也可以发一轮问卷，看有多少学生愿意在平台上预购下学期教材。这样你在学期初就能完成一个 BML 循环。"

### Q3.14 你怎么定义"学到了"？是数据达标还是用户反馈了什么？

**Story**: 用户说："看到群里有人问价、有教材上架，就算学到了吧。" 还是模糊，用户讨厌给自己设精确的标准。

**Demo**: Agent 很温和地追问："'有人问价'是个感觉，我需要你把它转成一个判断条件。比如，如果一周内群里有 30 本教材上架并且至少成交 5 单，就算验证了'学生愿意在专门平台交易'这个假设。如果上架数不到 10 本，就算学到了'推广大法不对'。定义一个清晰的'学到了'标准，实验结束后你才知道下一步是加大投入还是换方向。"

**Acceptance**: 用户能给出一个二元判断标准。"如果一周交易量达到 10 单，我就认为'学生愿意在专门平台交易'这个假设验证了。如果达不到，我就认为这个假设不成立，需要考虑转型。" 不再有中间地带。

**Error**: "学到了就是知道下一步做什么。" 这是循环论证。Agent 回："知道下一步做什么，是基于实验数据的判断。所以我问你。你期望看到什么数据？看到了就走哪条路，没看到又走哪条路？"

**Edge**: 用户说"学到了可能是'微信群模式推广不动'，这也是学习啊。" Agent 肯定："对！'学到了'不一定是好消息。所以你的标准里应该包含两种可能。如果一周成交 10 单，验证了平台模式可行。如果只成交 2 单，你学到了'武大学生可能没有想象的那么需要平台'，同时你学到一个新信息：也许问题不在需求，而在推广方式。"

### Q3.15 如果方向不对，你有哪些转型选项？

**Story**: 用户说："如果方向不对……到时候再看吧。可能换个学校推，或者换个品类。" 用户没有认真想过转型选项，把转型视为一个抽象概念。

**Demo**: Agent 鼓励用户提前思考："精益创业列出了 10 种转型类型。针对你的校园二手教材平台，最可能的转型选项有哪些？比如客户细分转型，从 C2C 平台转成 B2C（平台自己收书卖书）。或者价值获取转型，从交易佣金转成广告收益。你提前看看这些选项。如果现状不行，你最可能选哪个方向？"

**Acceptance**: 用户能从具体场景出发选出 2-3 个最可能的转型方向，并说出选择理由。比如"如果 C2C 撮合效率太低，我可能转 B2C。平台低价收书再卖出（价值获取转型）。如果教材品类太窄，我可能扩展到全品类校园二手（放大转型）。"

**Error**: "我到时候看数据再决定。" Agent 不放过："到时候看数据的最大问题是，数据本身不会告诉你往哪转。提前想好转型选项，你的实验设计会更聪明。你会注意到跟转型方向相关的信号。比如如果你提前想过'可能转成 B2C'，在 MVP 阶段你就会注意观察有没有学生愿意把书直接卖给你。"

**Edge**: 用户说"准备了 10 种转型也没用，我又不能同时做。" Agent 回："对，不能同时做。所以我让你选 2-3 个最可能的。不是让你执行，而是让你在实验中留意相关信号。这也是降低不确定性的一种方式。"

### Q3.16 你的资金跑道（runway）有多长？在钱花完前能试几次 pivot？

**Story**: 用户说："我自己投了 30 万，大概能撑 6 个月。如果还没起色，可能就得去找融资了。" 用户把资金跑道当成一个固定的时间线，但没有算过"能试几次"。

**Demo**: Agent 帮忙算了笔账："6 个月的资金跑道，如果每两周做一次实验（建群、推传单、撮合交易、收集数据），你可以试 12 次。但如果你每次实验都花一个月去做小程序开发，你只能试 6 次，而且如果第一个假设就错了，你两个月后才知道。所以问题不仅是'跑道多长'，更是'跑道能跑几圈'。你的 MVP 类型决定了你能跑多少圈。"

**Acceptance**: 用户能算出"跑道能跑几次实验"，而不仅仅是"跑道能跑几个月"。用户意识到加速循环比拉长跑道更重要。会说"如果我用微信群做 MVP、一周一个循环，我在钱花完前可以试 24 次。"

**Error**: "钱不够就去融。" Agent 提醒："在验证阶段融资越来越难。投资人更倾向投已验证模式。最好把你的跑道规划成'不需要融资也能完成验证'。你的 30 万如果花在快速实验上，足够你跑几十次了。验证完成后再融，估值更高，条件更好。"

**Edge**: 用户说"我的跑道是无限长，家里有矿。" Agent 回："资金充裕是好事，但'无限跑道'也有风险。缺乏紧迫感会降低学习速度。精益创业的一个核心观点是：时间和资金压力反而是好事，它们逼你找到最快验证的方法。你的优势是可以多试几次，但依然要快。"

### Q3.17 "转型或坚持"的评审会你打算多久开一次？

**Story**: 用户说："就我一个人创业，跟谁评审？有问题了再想呗。" 用户觉得一个人创业不需要评审流程。

**Demo**: Agent 先认可用户的处境，然后给出了单人评审的方案："一个人更要做评审。设好每周五下午的提醒，花 30 分钟对着你的数据问自己三个问题：这周群里有几本教材上架？成交了几单？从这个数据我学到了什么。学生到底需不需要这个平台？如果连续两周交易量都在 5 单以下，我应该坚持还是转型？一个人创业的最大风险就是没有人质问你。所以你要自己扮演那个质问者。"

**Acceptance**: 用户愿意设定一个固定的评审节奏（每周或每两周），并承诺在评审会上基于数据做决策，而不是凭感觉。用户明白"定期评审"不是官僚主义，而是给自己一个强制刹车机制。

**Error**: "一个月一次吧，但如果有重要数据随时可以看。" Agent 回："随时看数据的习惯很好，但固定频次的评审会有一个心理学作用。它让你在实验设计阶段就倒推：我必须在评审会前拿到数据。这个 deadline 压力反而会逼你把实验做小做快。"

**Edge**: 用户在评审中说"数据不太好，但我感觉还可以再试试。" Agent 回："你的感觉可能有道理。但把'感觉'写下来作为一个假设，设计下一个实验来验证它。比如'我感觉只要再推一周就会有人来了'。好，那下周的目标是多少上架量？我们一起设一个数字，下周评审会看。"

### Q3.18 你的核心指标如果达到目标，一年后能转化出多少收入？

**Story**: 用户说："赚钱的事还没想过，先验证了再说吧。" 典型的阶段跳跃。验证阶段只顾学习，忘了最终是为了判断商业可行性。

**Demo**: Agent 温和地引导："验证不只是为了学到东西，最终是为了判断这个生意能不能赚钱。比如你的 MVP 验证了每周 50 本上架、20% 成交率，就是 10 笔交易。假设每本教材平均佣金 10 元，周收入 100 元，一年大概 5000 元。这个数连你服务器和交通费都不够。但如果你把成交量放大到每天 50 单呢？算算看，按你设的预期指标，一年后能转化出多少收入？"

**Acceptance**: 用户能根据自己设定的核心指标做一道简单的乘法，算出一个"如果验证通过，一年后收入大概是多少"的数字。比如"如果每天 50 笔交易、每笔佣金 10 元、一年 250 个交易日，年收入就是 12.5 万。扣除人工和推广成本，毛利大概 5-6 万。这个规模还不足以全职做，但证明了模式可行。"

**Error**: "收入这个东西太远了，现在算不准。" Agent 回："当然算不准，但这不是财务预测，这是一个'如果……那么……'的推理。它帮你回答一个问题：如果学生的确愿意在这个平台交易，这个生意值得做吗？如果算出来的收入连你发传单的成本都覆盖不了，那即使验证了假设，这个模式也有问题。"

**Edge**: 用户说"我不是靠佣金赚钱，我是靠广告和增值服务。" 更复杂了。Agent 回："那你的计算链条是：活跃用户数到广告展示量到广告收入。你能给出预期吗？可以按最保守、最乐观、最可能三个场景各算一次。"

### Q3.19 把你的 MVP 阶段的实验数据，放大到 1000 个客户试试。

**Story**: 用户说："我们现在测试了 50 个学生用户，上架了 30 本教材、成交了 6 单，成交率 20%。放大到 1000 个学生……收入大概 1000*20%*10 元佣金 = 2000 元？好像不太多。" 用户在做放大计算，对自己的模式产生了怀疑。

**Demo**: Agent 肯定了用户的数学直觉，然后追问了成本端："收入算了，成本呢？注意，放大到 1000 个学生时，很多成本不是线性的。你现在人工撮合的方式在 50 个学生的微信群里没问题，但到了 1000 个学生，你可能需要客服、需要支付系统、需要处理退换货纠纷。你算算 1000 个学生时的预计成本，跟 2000 元收入比比，毛利是多少？如果毛利是负的，这个模式在规模上就不成立。"

**Acceptance**: 用户不仅算出了放大后的收入，也算出了放大后的成本结构，并理解了两者的关系。用户能得出一个"这个模式在 1000 个学生规模下是否健康"的判断。

**Error**: "放大后收入 2000 块，成本大概几百块吧，应该还行。" 这个"大概几百块"太模糊了。Agent 不放过："'大概'不够。做一次更具体的成本推演。1000 个学生需要几个客服处理纠纷？每本书的物流成本是多少？支付通道费占多少？把这些列出来，加总，再看毛利。"

**Edge**: 用户说"1000 个学生太远了，我们群里才 50 人。" 感觉被冒犯。Agent 回："不是让你现在就为 1000 个学生做准备，而是一种思维练习。验证阶段最容易犯的错误是'在小规模上验证了一个无法规模化的模式'。你现在人工撮合 50 个学生没问题，但到了 1000 个学生，你必须自动化。提前想想放大后的成本结构，可以帮你避免验证成功但规模化失败的尴尬。"
